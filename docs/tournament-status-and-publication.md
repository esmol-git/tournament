# Статусы турнира, публикация на сайте и правки после завершения

Кратко для администраторов организации и разработчиков.

## Два независимых понятия

1. **Статус жизненного цикла** (`DRAFT` → `ACTIVE` → `COMPLETED`, плюс `ARCHIVED` вручную) — отражает этап подготовки и проведения турнира.
2. **Публикация** (`published`) — виден ли турнир на **публичном** сайте (список турниров, прямые ссылки с `tid=…`, таблица, новости и т.д.).

Их можно комбинировать: например, турнир **в архиве**, но **опубликован** — страница по ссылке остаётся доступной для зрителей (удобно для прошедших сезонов).

## Статусы (автоматика)

| Статус | Когда так |
|--------|-----------|
| **DRAFT** | Пока у турнира **нет ни одного матча** в расписании. |
| **ACTIVE** | Как только появился **хотя бы один матч** (генерация календаря, шаблон, плей-офф, ручное расписание MANUAL). |
| **COMPLETED** | Все матчи турнира в финальном состоянии по протоколу: сыграны, завершены или отменены. |
| **ARCHIVED** | Только **вручную** из списка турниров («в архив»). Автоматика этот статус не снимает и не меняет. |

Ручной выбор статуса в форме турнира убран: значение подстраивается под матчи (см. `TournamentsService.syncTournamentLifecycleStatus`).

## Публикация (`published`)

- Включить сайт можно для турниров **не в черновике**: пока статус **DRAFT**, API **запрещает** `published: true` (ошибка с кодом `TOURNAMENT_PUBLISH_DRAFT_FORBIDDEN`).
- Для **ACTIVE**, **COMPLETED** и **ARCHIVED** публикация разрешена — в том числе для архивных турниров, если нужно оставить публичную страницу.
- При создании турнира он всегда создаётся как **DRAFT**; передать `published: true` при создании нельзя.

Публичный каталог турниров на сайте дополнительно фильтрует по статусу (черновики в общем списке не показываются), а прямая ссылка по `id` требует только `published === true`.

## Редактирование после «завершения»

### Протоколы матчей (счёт, события)

- **Не нужно** откатывать статус турнира с `COMPLETED` на `ACTIVE`, чтобы поправить ошибку в протоколе.
- Право править **уже закрытый** матч (PLAYED / FINISHED / CANCELED) есть у:
  - **администратора организации** (`TENANT_ADMIN`),
  - **администратора турнира** (`TOURNAMENT_ADMIN`),
  - **супер-администратора платформы** (`SUPER_ADMIN`), если работает в этом контексте.

Реализация: `MatchesService.canOverrideLockedProtocol` — для этих ролей не вызывается блокировка «завершённый матч нельзя изменять».

Остальные роли с доступом к протоколу (например, **REFEREE**) по-прежнему ограничены **до** финального закрытия матча.

### Карточка турнира (название, даты, формат, составы…)

- Редактирование **структуры** турнира в UI завязано на **DRAFT** (составы, группы без сыгранных матчей и т.д.) — это отдельное правило интерфейса, не статус «для протокола».
- Точечные изменения через API после старта сезона — по правилам конкретных эндпоинтов и ролей.

## Сводка для поддержки

| Задача | Действие |
|--------|----------|
| Показать турнир на сайте | Включить **публикацию**; статус не должен быть **DRAFT**. |
| Скрыть с сайта | Выключить **публикацию**. |
| Убрать из текущей работы | **Архив** (статус), публикация по желанию. |
| Исправить протокол после окончания | Достаточно прав **TENANT_ADMIN** / **TOURNAMENT_ADMIN** / **SUPER_ADMIN**; статус турнира не трогать. |

## Связанный код (ориентиры)

- Автостатусы: `backend/src/tournaments/tournaments.service.ts` — `syncTournamentLifecycleStatus`, создание турнира (`status: DRAFT`).
- Запрет публикации черновика: тот же сервис, `create` / `update`.
- Публичный доступ: `assertPublicTournament`, `listPublicByTenantSlug` и связанные методы.
- Протокол: `backend/src/matches/matches.service.ts` — `canOverrideLockedProtocol`, `updateProtocol`.
