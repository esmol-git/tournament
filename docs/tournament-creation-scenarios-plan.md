## 2. Два разных «формата» (не путать в UI)

| Термин в продукте | Поле в БД | Примеры |
|-------------------|-----------|---------|
| **Схема соревнования** | `Tournament.format` | одна группа, группы + плей-офф, только плей-офф, ручное расписание |
| **Формат игры на поле** | `Tournament.gameFormat` | 3×3 без вратарей, 5+1, 11×11 |

Покрытие поля — на **стадионе** (`Stadium.surfaceType` + уточнение).

## 3. Матрица сценариев создания турнира

Легенда: ✅ поддерживается · 🟡 частично · ❌ нет · 🔧 нужна доработка UX/регламента

### 3.1 Схема соревнования × набор команд

| Сценарий | Схема | Набор команд | Статус | Заметки |
|----------|-------|--------------|--------|---------|
| Лига одна группа | `SINGLE_GROUP` | MANUAL | ✅ | Круговая, автокалендарь |
| Лига с выходом в плей-офф | `GROUPS_PLUS_PLAYOFF` | MANUAL | ✅ | 2–4 группы |
| Кубок (олимпийка) | `PLAYOFF` | MANUAL | ✅ | 4/8/16… команд |
| Турнир выходного дня, сетка вручную | `MANUAL` | MANUAL | ✅ | Матчи по одному |
| Чемпионат с заявками клубов | любая | APPLICATIONS | ✅ | Окна приёма enforced |
| Фестиваль: много мелких турниров | — | — | 🟡 | Несколько турниров вручную, нет «родителя» |
| Сезон из нескольких этапов | — | — | ❌ | Нет multi-stage в одном турнире |

### 3.2 Аудитория и регламент поля

| Сценарий | Профиль | Формат игры | Покрытие | Статус |
|----------|---------|-------------|----------|--------|
| Детский U10, 5+1 | YOUTH | `FIVE_PLUS_ONE` | любое | ✅ | Возраст проверяется при заявке в состав |
| Мини-футбол 3×3 без вратаря | YOUTH | `THREE_V_THREE_NO_GK` | зал/синтетика | 🟡 |
| Взрослый любитель 8×8 | STANDARD | `EIGHT_V_EIGHT` | натуральный | ✅ |
| Профи 11×11 | STANDARD | `ELEVEN_V_ELEVEN` | натуральный | ✅ |
| Нестандарт (3+1 и т.д.) | любой | CUSTOM + note | любое | ✅ |

### 3.3 Состав и заявки

| Сценарий | Статус | Заметки |
|----------|--------|---------|
| Организатор вносит заявку клуба + Excel | ✅ | |

## 6. Чеклист «покрыты ли все типичные турниры»

| Тип турнира | Можно завести сейчас? | Комфортно довести до финала? |
|-------------|----------------------|------------------------------|
| Детский круговой 1 группа | ✅ | 🟡 (возраст, протокол) |
| Детский группы + плей-офф | ✅ | 🟡 |
| Кубок на 8/16 команд | ✅ | ✅ |
| Однодневный фестиваль (несколько возрастов) | 🟡 | 🟡 (N отдельных турниров) |
| Лига на сезон | ✅ | 🟡 |
| Товарищеские матчи без таблицы | ✅ MANUAL | ✅ |
| Турнир только с заявками | 🟡 | 🟡 (enforcement) |
| Турнир на нескольких площадках | ✅ | 🟡 (pitchCount не в планировщике) |

**Вывод:** основные **структурные** сценарии покрыты. Критичное слабое место на дистанции — **связность справочников с регламентом** (см. §7), а не отсутствие формата турнира в принципе.

## 7. Критические пробелы справочников (продуктовая модель)

**Полное проектирование:** [`competition-edition-design.md`](./competition-edition-design.md) — сущность **«Зачёт»**, аудитория (дети / взрослые любители / open), `EligibilityPolicy`, scope санкций.

**Диагноз:** часть справочников заведена как «поля в форме турнира», без сквозной сущности **программы соревнований**. Ментальная модель заточена под детские турниры; для взрослых лиг те же поля (возраст, категория) вводят в заблуждение.

### 7.1 Пробел → целевое поведение

| # | Пробел сейчас | Что ждёт организатор | Целевое решение |
|---|---------------|----------------------|-----------------|
| 1 | Сезон и соревнование — только метки | Один сезон = одна лига, общая логика | Сущность **«Зачёт»** (сезон + тип соревнования): группа турниров, общие настройки, опционально сводная таблица |
| 2 | Возраст vs категория команды дублируются | Один регламент допуска | Категория **наследует** возрастную группу ИЛИ одно поле «регламент допуска»; предупреждение при расхождении |
| 3 | Санкции только в турнире | Жёлтые по кубку / по сезону | `sanctionScope`: турнир \| зачёт \| сезон (настройка на зачёте или турнире) |
| 4 | Стадион не связан с форматом игры | Подсказка при несовпадении | Мягкое предупреждение: покрытие стадиона vs `gameFormat` турнира |
| 5 | Нет «родителя» для фестиваля | 5 возрастов = одна программа | Зачёт объединяет N турниров; общий сезон, бренд, документы |
| 6 | Нет онбординга справочников | Порядок «сначала возраст, потом импорт» | Мастер первого запуска организации + шаг «контекст» в создании турнира |

### 7.2 Предлагаемая сущность: «Зачёт» (`CompetitionEdition`)

См. детали, схему Prisma, сценарии по аудитории и фазы E3a–E5 в [`competition-edition-design.md`](./competition-edition-design.md).

Кратко:

```
Season + Competition + audience → CompetitionEdition (Зачёт)
  ├── EligibilityPolicy (единый регламент допуска)
  ├── sanctionScope
  └── tournaments[] (1..N)
```

### 7.3 Фазы внедрения (приоритет)

**Фаза E1 — быстрые победы (1–2 недели, без новой сущности)**

- [ ] E1.1 Предупреждения: возраст турнира ≠ категория команды при заявке
- [ ] E1.2 `TeamCategory.ageGroupId` (опционально) — копирование годов рождения
- [ ] E1.3 UI: сезон/соревнование подписать «для фильтра и отчётов» + tooltip
- [ ] E1.4 Предупреждение: стадион vs формат игры
- [ ] E1.5 Чеклист онбординга на дашборде: возраст → категории → команды → турнир

**Фаза E2 — scope санкций (2–3 недели)**

- [ ] E2.3 UI в настройках турнира / зачёта

**Фаза E3 — сущность «Зачёт»** → см. **E3a/E3b** в [`competition-edition-design.md`](./competition-edition-design.md)

**Фаза E5 — EligibilityPolicy** (единый регламент вместо дублирования age/category)

**Фаза E4 — мастер и упрощение UX (параллельно E2–E3)**

- [ ] E4.1 Шаг «Контекст» в создании турнира (сезон, зачёт, возраст, формат игры)
- [ ] E4.2 Пресет «Детская лига» — автозаполнение справочников

### 7.4 Что можно упростить без кода (сейчас)

- Не заводить **и** возраст на команде, **и** категорию с теми же годами — категория для пола/особых правил, возраст — на турнире.
- Сезон/соревнование использовать только когда реально нужна отчётность.
- Один кубок = один турнир; многоэтапный кубок пока = один турнир с группами + плей-офф, а не два турнира с одной меткой.
