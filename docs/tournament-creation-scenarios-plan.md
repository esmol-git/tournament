## 2. Два разных «формата» (не путать в UI)

| Термин в продукте | Поле в БД | Примеры |
|-------------------|-----------|---------|
| **Схема соревнования** | `Tournament.format` | одна группа, группы + плей-офф, только плей-офф, ручное расписание |
| **Формат игры на поле** | `Tournament.gameFormat` | 3×3 без вратарей, 5+1, 11×11 |

Покрытие поля — на **стадионе** (`Stadium.surfaceType` + уточнение).

## 3. Матрица сценариев создания турнира

Легенда: ✅ поддерживается · 🟡 частично · ❌ нет · 🔧 нужна доработка UX/регламента

### 3.1 Схема соревнования × набор команд

| Сценарий | Схема | Набор команд | Статус | Заметки |
|----------|-------|--------------|--------|---------|
| Лига одна группа | `SINGLE_GROUP` | MANUAL | ✅ | Круговая, автокалендарь |
| Лига с выходом в плей-офф | `GROUPS_PLUS_PLAYOFF` | MANUAL | ✅ | 2–4 группы |
| Кубок (олимпийка) | `PLAYOFF` | MANUAL | ✅ | 4/8/16… команд |
| Турнир выходного дня, сетка вручную | `MANUAL` | MANUAL | ✅ | Матчи по одному |
| Чемпионат с заявками клубов | любая | APPLICATIONS | 🟡 | Заявки есть, окна приёма не enforced |
| Фестиваль: много мелких турниров | — | — | 🟡 | Несколько турниров вручную, нет «родителя» |
| Сезон из нескольких этапов | — | — | ❌ | Нет multi-stage в одном турнире |

### 3.2 Аудитория и регламент поля

| Сценарий | Профиль | Формат игры | Покрытие | Статус |
|----------|---------|-------------|----------|--------|
| Детский U10, 5+1 | YOUTH | `FIVE_PLUS_ONE` | любое | 🟡 | Формат в форме; возраст по birthDate не проверяется |
| Мини-футбол 3×3 без вратаря | YOUTH | `THREE_V_THREE_NO_GK` | зал/синтетика | 🟡 |
| Взрослый любитель 8×8 | STANDARD | `EIGHT_V_EIGHT` | натуральный | ✅ |
| Профи 11×11 | STANDARD | `ELEVEN_V_ELEVEN` | натуральный | ✅ |
| Нестандарт (3+1 и т.д.) | любой | CUSTOM + note | любое | ✅ |

### 3.3 Состав и заявки

| Сценарий | Статус | Пробел |
|----------|--------|--------|
| Организатор вносит заявку клуба + Excel | ✅ | |
| Организатор набирает состав, подтверждает | ✅ | |
| Дисквалификация на турнир | ✅ | Только в DRAFT; не в протоколе |
| Клуб сам подаёт заявку/состав | ❌ | Осознанно organizer-centric |
| Проверка возраста vs ageGroup | ❌ | |
| Протокол из турнирного roster | ❌ | Берётся каталог команды |

### 3.4 Инфраструктура при создании

| Что | В мастере создания | После создания | Статус |
|-----|-------------------|----------------|--------|
| Команды | шаг enrollment / participants | вкладка команды | ✅ / 🟡 |
| Сезон, соревнование, возраст | полная форма, не в wizard | ✅ | 🟡 wizard не ведёт |
| Стадионы, судьи | полная форма | вкладка инфраструктура | 🟡 |
| Шаблон турнира | отдельно | templates | ✅ |
| Мастер запуска (5 шагов) | после создания | `[id]` launch | ✅ |

## 4. Пробелы мастера создания (приоритет UX)

Сейчас wizard: **О турнире → Схема → Участники → Обзор**.

| # | Улучшение | Зачем |
|---|-----------|-------|
| W1 | Переименовать шаг «Формат» → «Схема соревнования» | Не путать с форматом игры |
| W2 | Пресеты: «Детский фестиваль 5+1», «Кубок на выходных», «Лига» | Быстрый старт без 20 полей |
| W3 | Шаг «Контекст»: сезон + возрастная группа + gameFormat | Типичный детский сценарий в одном месте |
| W4 | Подсказка по покрытию: «выберите стадионы после создания» | Связь stadium ↔ surfaceType |
| W5 | Валидация: APPLICATIONS без maxTeams / дат — предупреждение | Меньше тупиков |
| W6 | Сводка в обзоре: схема + игра + профиль + лимиты состава | Уже частично |

## 5. Дорожная карта доработок (после охвата)

### Фаза A — закрыть тупики без новых экранов
1. Enforcement окон заявок (`registrationOpensAt` / `closesAt`)
2. Протокол и публичный состав из `TournamentTeamPlayer`
3. Блок дисквалифицированных в протоколе
4. Проверка birthDate vs `ageGroup` при roster

### Фаза B — регламент и операционка
5. Судьи на матч (главный + помощники)
6. Уведомление «матч скоро»
7. Карточки → автобан N матчей
8. Технические результаты (неявка)

### Фаза C — публичка и отчёты
9. Единая логика `publishedOnPublic` для общей таблицы
10. PDF протокол / таблица
11. ICS по команде / стадиону
12. Трансляции (сейчас заглушка)

### Фаза D — тестирование
- Чеклист сценариев из §3 (ручной + автотесты API)
- Моделирование: отмена матча, выход команды, ничья в плей-офф, перенос, DQ mid-tournament

## 6. Чеклист «покрыты ли все типичные турниры»

| Тип турнира | Можно завести сейчас? | Комфортно довести до финала? |
|-------------|----------------------|------------------------------|
| Детский круговой 1 группа | ✅ | 🟡 (возраст, протокол) |
| Детский группы + плей-офф | ✅ | 🟡 |
| Кубок на 8/16 команд | ✅ | ✅ |
| Однодневный фестиваль (несколько возрастов) | 🟡 | 🟡 (N отдельных турниров) |
| Лига на сезон | ✅ | 🟡 |
| Товарищеские матчи без таблицы | ✅ MANUAL | ✅ |
| Турнир только с заявками | 🟡 | 🟡 (enforcement) |
| Турнир на нескольких площадках | ✅ | 🟡 (pitchCount не в планировщике) |

**Вывод:** основные **структурные** сценарии покрыты. Слабое место — **связность** (заявка → состав → протокол → публичка) и **детский регламент**, а не отсутствие формата турнира в принципе.
