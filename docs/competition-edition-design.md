# Проектирование: «Зачёт» (Competition Edition)

Документ фиксирует целевую модель для **широкой аудитории** (дети, любители-взрослые, более серьёзный взрослый футбол), а не только сценарий «детская лига в голове организатора».

Связанные файлы:
- [`tournament-creation-scenarios-plan.md`](./tournament-creation-scenarios-plan.md) — §7 пробелы и фазы E1–E4
- [`admin-docs-maintenance.md`](./admin-docs-maintenance.md) — user guide после внедрения

---

## 1. Диагноз: почему «детская модель вшита в ядро»

Сейчас регламент **размазан** по сущностям без общего родителя:

| Поле / справочник | Где живёт | Смысл для детей | Для взрослых |
|-------------------|-----------|-----------------|--------------|
| `AgeGroup` | турнир, команда | Центрально | Часто не нужен |
| `TeamCategory` | команда | Пол + годы рождения | Скорее «дивизион» (A/B, 35+) |
| `eligibilityProfile` YOUTH/STANDARD | турнир | Грубый переключатель | Не описывает любительский/профи |
| `seasonId`, `competitionId` | турнир | Ожидается «лига» | Только метки |
| Карточки, DQ | турнир | Ок для одного кубка | Нужен scope по зачёту/сезону |
| `gameFormat`, стадион | турнир / стадион | 5+1, зал | 8×8, 11×11, газон |

**Итог:** система позволяет провести взрослый турнир, но **ментальная модель UI и справочников** заточена под детство. На дистанции организатор взрослой лиги не понимает, зачем возрастная группа; организатор детского фестиваля — почему нет «одной лиги» на сезон.

## 2. Целевая иерархия

```
Организация (Tenant)
  └── Справочники (сезон, тип соревнования, стадионы, судьи, дивизионы…)
  └── Зачёт (CompetitionEdition)          ← НОВАЯ сущность, центр регламента
        ├── Регламент допуска (наследуемый)
        ├── Scope санкций (карточки, DQ)
        ├── Сезон + тип соревнования
        └── Турнир 1..N (этап / возраст / дивизион)
              └── Матчи → протокол → таблица
```

### 2.1 Что такое «Зачёт»

**Зачёт** — программа соревнований с **единым регламентом и идентичностью** для участников и сайта.

Примеры:

| Зачёт | Турниры внутри |
|-------|----------------|
| Чемпионат города 2025/26 | U10, U12, U14 (3 турнира) |
| Любительская лига «Премьер» весна 2026 | Одна группа + плей-офф (1 турнир) или Дивизион A + B (2 турнира) |
| Кубок организации 2026 | Групповой этап + плей-офф (1 турнир с форматом) или 2 этапа (2 турнира) |
| Корпоративный турнир выходного дня | 1 турнир без зачёта (legacy) |

### 2.2 Что остаётся «Турниром»

**Турнир** — операционная единица: команды, группы, календарь, матчи, таблица, публикация.

- Может существовать **без зачёта** (`editionId = null`) — разовый турнир, как сейчас.
- В зачёте — **наследует** регламент и санкции, может **переопределить** отдельные поля (явно в UI: «отличается от зачёта»).

## 3. Аудитория: не только дети

Новый enum **`CompetitionAudience`** (на уровне зачёта, с наследованием на турнир):

| Значение | Кто | Типичный регламент | Возраст / категории |
|----------|-----|--------------------|---------------------|
| `YOUTH` | Дети и юноши | Годы рождения, пол, заявочный состав | AgeGroup + TeamCategory |
| `ADULT_AMATEUR` | Взрослые любители | Дивизион силы, без жёсткого возраста | TeamCategory как «дивизион»; возраст опционально (35+, 40+) |
| `ADULT_COMPETITIVE` | Взрослые соревновательные лиги | Ростер, дисциплина, возможно лимит гостей | Минимальные проверки возраста |
| `OPEN` | Любой / смешанный | Минимум ограничений | Без ageGroup по умолчанию |

`TournamentEligibilityProfile` (YOUTH/STANDARD) **заменяется постепенно** на `CompetitionAudience` + **`EligibilityPolicy`** (см. §5). На переходный период — маппинг: YOUTH → audience YOUTH, STANDARD → ADULT_AMATEUR.

## 4. Модель данных (v1)

### 4.1 `CompetitionEdition`

```prisma
model CompetitionEdition {
  id          String   @id @default(cuid())
  tenantId    String
  name        String                    // «Чемпионат города 2025/26»
  slug        String
  description String?

  seasonId       String                 // обязателен для зачёта в v1
  competitionId  String                 // тип: чемпионат, кубок, лига

  audience    CompetitionAudience @default(OPEN)

  startsAt    DateTime?
  endsAt      DateTime?

  /// Наследуемый регламент допуска (см. EligibilityPolicy)
  eligibilityPolicyId String?

  /// Санкции: в рамках одного турнира / всего зачёта / всего сезона
  sanctionScope SanctionScope @default(EDITION)

  cardAutoBanEnabled Boolean @default(false)
  redCardBanMatches Int @default(1)
  yellowAccumulationThreshold Int @default(2)
  yellowAccumulationBanMatches Int @default(1)
  technicalWinGoalsFor Int @default(3)
  technicalWinGoalsAgainst Int @default(0)

  published   Boolean @default(false)
  status      EditionStatus @default(DRAFT)  // DRAFT | ACTIVE | ARCHIVED

  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt

  tenant      Tenant @relation(...)
  season      Season @relation(...)
  competition Competition @relation(...)
  eligibilityPolicy EligibilityPolicy? @relation(...)
  tournaments Tournament[]
}
```

### 4.2 Изменения `Tournament`

```prisma
model Tournament {
  // ... существующие поля ...

  /// null = автономный турнир (как сейчас)
  editionId   String?
  edition     CompetitionEdition? @relation(...)

  /// inherit | override — как турнир относится к регламенту зачёта
  regulationMode TournamentRegulationMode @default(INHERIT)

  // seasonId, competitionId — остаются для обратной совместимости;
  // при editionId заполняются из зачёта (триггер/сервис), UI скрывает дубли
}
```

### 4.3 Enums

```prisma
enum CompetitionAudience {
  YOUTH
  ADULT_AMATEUR
  ADULT_COMPETITIVE
  OPEN
}

enum SanctionScope {
  TOURNAMENT   // только этот турнир (разовый кубок)
  EDITION      // все турниры одного зачёта (кубок из 2 этапов)
  SEASON       // все турниры сезона (редко; явное предупреждение в UI)
}

enum TournamentRegulationMode {
  INHERIT      // регламент и санкции с зачёта
  OVERRIDE     // свой ageGroup, карточки и т.д. (подсветка в UI)
}

enum EditionStatus {
  DRAFT
  ACTIVE
  ARCHIVED
}
```

## 5. Единый регламент допуска: `EligibilityPolicy`

Вместо дублирования AgeGroup + TeamCategory + полей турнира — **политика**, которую можно привязать к зачёту, турниру или категории команды.

```prisma
model EligibilityPolicy {
  id        String @id @default(cuid())
  tenantId  String
  name      String?                   // «U12 мальчики», «Мужчины Дивизион A»

  requireBirthDate Boolean @default(false)
  minBirthYear     Int?
  maxBirthYear     Int?
  allowedGenders   PlayerGender[] @default([])

  rosterMinPlayers Int?
  rosterMaxPlayers Int?
  rosterDeadlineAt DateTime?        // обычно на турнире, не на зачёте

  /// Опциональная ссылка на справочник для UI
  ageGroupId       String?

  tenant Tenant @relation(...)
  ageGroup AgeGroup? @relation(...)
}
```

**Правила применения (порядок проверки):**

1. Политика **турнира** (если `regulationMode = OVERRIDE` или автономный турнир)
2. Иначе политика **зачёта**
3. Плюс **TeamCategory** команды (если задана) — должна быть **совместима** с политикой турнира/зачёта, иначе предупреждение или ошибка

**Связь TeamCategory → EligibilityPolicy** (или `ageGroupId`):

- Детская команда: категория ссылается на ту же политику / ageGroup, что зачёт U12
- Взрослая команда: категория «Мужчины A» — политика без годов рождения

## 6. Санкции и scope

Состояние санкций хранится на **`EditionTeamPlayer`** или остаётся на `TournamentTeamPlayer` с **агрегацией по scope** при расчёте.

**v1 (проще):** оставить `TournamentTeamPlayer.suspendedMatchesRemaining`, но пересчёт карточек смотрит матчи:

| `sanctionScope` | Матчи для подсчёта жёлтых/красных |
|-----------------|-----------------------------------|
| `TOURNAMENT` | только `tournamentId` |
| `EDITION` | все турниры с `editionId` |
| `SEASON` | все турниры с `seasonId` (осторожно: разные зачёты одного сезона) |

Блок в протоколе — по **турнирному** составу, но флаг «отбывает бан» считается по scope.

## 7. Справочники после введения зачёта

| Справочник | Роль |
|------------|------|
| **Сезон** | Измерение времени; обязателен для зачёта; фильтры |
| **Соревнование** | Тип (чемпионат, кубок); обязателен для зачёта |
| **Зачёт** | **Продуктовая сущность**, не справочник-одна строка — отдельный раздел админки |
| **Возрастная группа** | Справочник; используется внутри `EligibilityPolicy` для YOUTH |
| **Категория команды** | Дивизион / пол / возраст на уровне клуба; ссылка на policy |
| **Стадион, судья** | Без изменений; пул на турнир |

**Сезон и соревнование перестают вешаться напрямую на турнир в UI** (если турнир в зачёте) — только на зачёт.

## 8. Сценарии по аудитории

### 8.1 Детский фестиваль (5 возрастов)

1. Сезон «Лето 2026», соревнование «Фестиваль»
2. Зачёт «Фестиваль Лето 2026», audience `YOUTH`, sanctionScope `EDITION` (если общие правила карточек)
3. Турниры: U10, U12, U14, U16, U18 — каждый с `INHERIT` + override только `ageGroup`/policy
4. Публичный сайт: страница зачёта со списком турниров

### 8.2 Взрослая любительская лига (без возраста)

1. Зачёт «Премьер-лига весна 2026», audience `ADULT_AMATEUR`
2. EligibilityPolicy без birth years; roster max опционально
3. Один турнир, группа + плей-офф
4. TeamCategory: «Дивизион 1», «Дивизион 2» — без minBirthYear

### 8.3 Разовый корпоративный кубок

1. Турнир без зачёта, sanctionScope `TOURNAMENT`
2. audience `OPEN`, без ageGroup
3. Поведение как сейчас

### 8.4 Кубок в два этапа (группа + плей-офф как отдельные турниры)

1. Зачёт «Кубок 2026», sanctionScope `EDITION` — жёлтые из группы тянутся в плей-офф
2. Турнир «Групповой этап», турнир «Плей-офф»
3. Продвижение команд — ручное или через существующий плей-офф в одном турнире (продуктово предпочтительнее 1 турнир с форматом; 2 турнира — для особых регламентов)

## 9. UI / админка (эскиз)

### Новый раздел: «Зачёты» (`/admin/editions` или под «Турниры»)

- Список зачётов по сезону
- Карточка зачёта: регламент, санкции, список турниров, публикация
- «Создать турнир в зачёте» / «Привязать существующий»

### Создание турнира

- Шаг 1: **Автономный** или **В зачёте**
- Если в зачёте — выбор зачёта, audience и policy подставляются
- Поля season/competition/age скрыты при INHERIT

### Онбординг организации

- Вопрос: «Кого в основном обслуживаете?» → пресет справочников
- Дети: age groups + категории
- Взрослые: дивизионы без возраста, gameFormat 8×8/11×11

---

## 10. Миграция и обратная совместимость

1. **Фаза 0:** таблица `CompetitionEdition`, nullable `Tournament.editionId` — без обязательного использования
2. **Фаза 1:** CRUD зачёта, привязка турниров; UI
3. **Фаза 2:** `EligibilityPolicy`, проверки в roster; связь TeamCategory
4. **Фаза 3:** `sanctionScope` пересчёт карточек
5. **Опционально:** скрипт — пары `(seasonId, competitionId)` → автозачёт для старых данных

Турниры без `editionId` работают **без изменений**.
